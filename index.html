BLOC 1: Fonaments de l'asincronia i AJAX

Pregunta 1: Què vol dir que JavaScript és "monofil" (single-threaded)?

Que només pot executar una instrucció alhora en un únic fil d’execució (call stack). No pot fer dues operacions de processament de dades simultànies.

console.log("Primer");
console.log("Segon");
// El fil d'execució no pot passar al segon fins que acabi el primer.

Pregunta 2: Quina és la diferència principal entre execució síncrona i asíncrona?

Síncrona: el codi s’executa línia a línia, bloquejant fins acabar cada tasca. Asíncrona: permet delegar tasques (com demanar dades) i continuar executant la resta del codi sense esperar que la tasca delegada acabi.

// Síncron: Espera 3 segons bloquejat
console.log("Inici");
esperar3Segons();
console.log("Fi");

// Asíncron: No es bloqueja
console.log("Inici");
setTimeout(() => console.log("Tascompletada"), 3000);
console.log("Fi"); // S'executa immediatament després de l'Inici

Pregunta 3: Què és un "callback"?

Una funció que es passa com a argument a una altra funció i s’executa més tard, normalment quan una operació asíncrona ha acabat.

function processarSalutacio(nom, callback) {
 console.log("Hola " + nom);
 callback();
}

processarSalutacio("Dani", () => {
 console.log("El callback s'ha executat!");
});

Pregunta 4: Què s'entén per "Callback Hell"?

Codi amb múltiples callbacks imbricats un dins de l'altre, fent que sigui molt difícil de llegir, entendre i mantenir (també anomenat "Pyramid of Doom").

pas1(() => {
 pas2(() => {
  pas3(() => {
   pas4(() => {
    console.log("Finalment!");
   });
  });
 });
});

Pregunta 5: Per què el codi síncron pot ser un perill per a la UI d'un navegador?

Perquè si una tasca síncrona triga molt (com un bucle infinit o una petició pesada), bloqueja el fil principal i congela la interfície. L'usuari no pot clicar botons ni fer scroll.

// AQUEST CODI CONGELA EL NAVEGADOR:
btn.addEventListener("click", () => {
 while(true) {
  // Bucle infinit: la UI no respondrà mai més
 }
});

Pregunta 6: Què vol dir que una funció és "non-blocking"?

Que quan s'executa, inicia l'operació (com demanar dades a un servidor) i retorna el control al programa immediatament, sense bloquejar el fil principal mentre espera la resposta.

// setTimeout és non-blocking
setTimeout(() => console.log("Han passat 2s"), 2000);
console.log("Jo no m'he bloquejat!");

Pregunta 7: Quin mètode clàssic de JS serveix per executar un codi després d'un temps determinat?

El mètode clàssic és setTimeout().

setTimeout(() => {
 console.log("Això s'executa als 5 segons");
}, 5000);

Pregunta 8: En l'asincronia, què és l'"Event Loop"?

És el mecanisme que gestiona l’execució del codi asíncron. Comprova constantment si el Call Stack està buit; si ho està, agafa el primer callback de la cua (Callback Queue) i el mou al Call Stack per executar-lo.

console.log("1");
setTimeout(() => console.log("2"), 0);
console.log("3");

// Ordre: 1, 3 i finalment el 2 via Event Loop.

Pregunta 9: Cert o Fals: L'asincronia serveix per fer dues coses exactament al mateix mil·lisegon en un sol fil.

Fals. JavaScript segueix sent monofil. El que fa l'asincronia és "intercalar" tasques. Mentre esperem una resposta externa, executem altres coses, però mai dues línies de codi JS al mateix instant real.

Pregunta 10: Com s'anomena la pila on es guarden les funcions que esperen a ser executades després d'una tasca asíncrona?

S'anomena Callback Queue (o Task Queue). Les promeses fan servir una cua prioritària anomenada Microtask Queue.

Pregunta 11: Què és l'objecte XMLHttpRequest (XHR) i quin era el seu paper fonamental fa uns anys?

És un objecte de l'API del navegador que permet fer peticions HTTP de manera asíncrona. El seu paper va ser fonamental per permetre actualitzar parts d'una web sense haver de recarregar la pàgina sencera.

const xhr = new XMLHttpRequest();
xhr.open("GET", "https://api.exemple.com/dades");
xhr.send();

Pregunta 12: Veritat o Mentida: Tot i dir-se XMLHttpRequest, aquest objecte només permet enviar i rebre dades en format XML.

Mentida. Tot i el seu nom, pot treballar amb JSON, text pla, HTML i altres formats. Avui dia el format més comú és JSON.

xhr.onload = function() {
 const dades = JSON.parse(xhr.responseText);
 console.log(dades);
};

Pregunta 13: Com que l'XHR no funcionava amb Promeses, quin esdeveniment s'havia d'escoltar per saber si el servidor ens havia respost?

S'havia d'escoltar l'esdeveniment onreadystatechange (o el més modern onload).

xhr.onreadystatechange = function() {
 // Es crida cada vegada que canvia l'estat
};

Pregunta 14: En una petició XHR clàssica, què significava comprovar que la propietat readyState === 4?

Significa que l'operació s’ha completat correctament i la resposta del servidor ha estat rebuda totalment.

if (xhr.readyState === 4 && xhr.status === 200) {
 console.log("Tot correcte!");
}

Pregunta 15: Quin era el principal desavantatge d'escriure codi amb XMLHttpRequest comparat amb la Fetch API moderna?

Té una sintaxi molt més complexa, verbosa i difícil de seguir. A més, no suporta Promeses de forma nativa, cosa que obliga a fer servir molts callbacks i dificulta la gestió d'errors.

// XHR és molt llarg per a una simple petició:
const xhr = new XMLHttpRequest();
xhr.open("GET", url);
xhr.onreadystatechange = function() { ... };
xhr.send();

Pregunta 16: Què signifiquen les sigles AJAX?

Asynchronous JavaScript And XML. És la tècnica de carregar dades en segon pla.

Pregunta 17: Cal recarregar la pàgina sencera quan fem servir AJAX?

No. Aquest és precisament el seu objectiu: actualitzar només una part de la interfície amb les noves dades rebudes.

fetch('/nova-noticia')
 .then(res => res.text())
 .then(html => {
  document.getElementById('noticies').innerHTML += html;
 });

Pregunta 18: Què és la Fetch API (definició bàsica)?

És una interfície moderna i nativa de JavaScript per fer peticions HTTP. A diferència d'XHR, està totalment basada en Promeses.

fetch('https://api.test.com/user')
 .then(response => console.log("Resposta rebuda"));

Pregunta 19: Per què fetch() és millor que l'antic XMLHttpRequest?

Perquè el codi és molt més net, llegible i curt. A més, en estar basat en Promeses, permet fer servir la sintaxi async/await, que fa que el codi asíncron sembli gairebé síncron.

// Molt més net que XHR:
const res = await fetch(url);
const data = await res.json();

Pregunta 20: Quina dada retorna fetch() en primera instància quan l'executem?

Retorna una Promise que, quan es resol, conté un objecte de tipus Response. Aquest objecte conté informació sobre la resposta (status, headers), però no les dades directament.

const promesa = fetch(url);
console.log(promesa); // Imprimeix: Promise {<pending>}

BLOC 2: El motor modern - Promeses i Async/Await

Pregunta 1: Què és una Promise en JavaScript?

Una Promise (promesa) és un objecte que representa l'eventual finalització (o el fracàs) d'una operació asíncrona i el seu valor resultant. És com un "val" per a un valor que encara no coneixem, però que rebrem en el futur.

const promesa = new Promise((resolve, reject) => {
 // L'operació asíncrona passa aquí dins
});

Pregunta 2: Quins són els 3 estats d'una promesa?

  • Pending (Pendent): Estat inicial. La promesa encara està en procés i no s'ha resolt ni rebutjat.
  • Fulfilled (Complerta / Resolta): L'operació ha acabat correctament i retorna un resultat.
  • Rejected (Rebutjada): L'operació ha fallat i retorna un error o motiu del rebuig.

Pregunta 3: Què vol dir que una promesa està "Pending"?

Una Promise en estat pending vol dir que encara està pendent de resoldre's. És a dir: ni ha acabat bé (fulfilled) ni ha fallat (rejected).

const promesa = new Promise((resolve, reject) => {
 setTimeout(() => {
  resolve("Fet!");
 }, 2000);
});

// Què passa?
// just crear-la → pending
// després de 2 segons → fulfilled

Pregunta 4: Com es passa de "Pending" a un estat d'èxit (Fulfilled)?

Es passa a l'estat Fulfilled quan cridem la funció interna resolve(valor). Aquesta funció es passa com a primer argument al constructor de la promesa i, en executar-se, indica que l'operació ha acabat correctament.

const guanyarLoteria = new Promise((resolve) => {
 setTimeout(() => {
  resolve("Has guanyat 1000€!"); // Passem a estat Fulfilled
 }, 1000);
});

Pregunta 5: Quin mètode s'utilitza per recollir el resultat d'una promesa quan va bé (.then)?

S'utilitza el mètode .then(). Aquest mètode rep una funció callback que s'executa automàticament quan la promesa es resol.

guanyarLoteria.then((premi) => {
 console.log(premi); // Imprimeix: "Has guanyat 1000€!"
});

Pregunta 6: Com es gestiona un error en una promesa sense fer servir async/await (.catch)?

Per gestionar aquests errors es fa servir molt el catch(). Com funciona? Tenim dos elements, el .then (gestiona el resultat correcte) i el .catch (captura qualsevol error que passi en qualsevol lloc de la cadena).

Promise.resolve(10)
 .then(num => {
  if (num > 5) {
   throw new Error('Número massa gran');
  }
  return num;
 })
 .then(result => {
  console.log(result);
 })
 .catch(err => {
  console.error(err.message);
 });

Pregunta 7: Pots encadenar diversos .then()? Si Per què serveix?

Cada .then() retorna una nova Promise, així que pots seguir encadenant-ne tantes com vulguis. Quan acaba un comença l'altre. Serveix per processar dades pas a pas (cada pas transforma o utilitza el resultat anterior), evitar el "callback hell", i controlar fluxos asíncrons.

fetch('/api/data')
 .then(response => response.json())
 .then(data => {
  console.log(data);
  return data.items;
 })
 .then(items => {
  console.log(items);
 });

login(user)
 .then(session => getProfile(session))
 .then(profile => loadDashboard(profile))
 .catch(error => console.error(error));

Pregunta 8: Què retorna sempre un .then()?

Un .then() sempre retorna una nova promesa. Això és el que permet l'encadenament (chaining). Si la funció dins del then() retorna un valor simple, aquest valor s'embolcalla automàticament en una promesa resolta.

guanyarLoteria
 .then(valor => valor.toUpperCase()) // Retorna promesa amb text en majúscules
 .then(textGran => console.log(textGran)); // Imprimeix el resultat de l'anterior

Pregunta 9: Per què serveix el mètode Promise.all()?

Serveix per gestionar múltiples promeses simultàniament. Rep un array de promeses i retorna una única promesa que es resol quan totes les promeses de l'array s'han completat amb èxit. Si una sola falla, tota l'operació falla.

const p1 = Promise.resolve("Pa");
const p2 = Promise.resolve("Formatge");

Promise.all([p1, p2]).then((compra) => {
 console.log(`Ja tinc el ${compra[0]} i el ${compra[1]}`);
 // "Ja tinc el Pa i el Formatge"
});

Pregunta 10: Què passa si una de les promeses dins de Promise.all() falla?

Si una de les promeses falla el Promise.all() farà un reject, que vol dir això, retorna l'error de la primera promesa, però ignorarà la resta de codi.

Pregunta 11: Què fa la paraula clau async abans d'una funció?

1. Fa que la funció sempre retorni una Promise. Encara que tu retornis un valor "normal", JavaScript el converteix automàticament en una Promise.
2. Permet usar await dins la funció. Només pots utilitzar await dins d'una funció async.

async function exemple() {
 return 42;
}

// És equivalent a:
function exemple() {
 return Promise.resolve(42);
}

async function getData() {
 const response = await fetch('/api/data');
 const data = await response.json();
 return data;
}

Pregunta 12: Què retorna sempre una funció marcada amb async?

Quan una funció està marcada amb async, sempre torna una promesa.

async function saludar() {
 return "Hola";
}
// saludar() devuelve Promise {<fulfilled>: "Hola"}
saludar().then(console.log); // Imprime: "Hola"

Pregunta 13: On es pot fer servir la paraula await?

La paraula clau await només es pot fer servir dins de funcions marcades amb la paraula async (o en el nivell superior de mòduls JavaScript, conegut com top-level await).

async function sopar() {
 // Només podem fer servir await aquí dins perquè la funció és 'async'
 const plat = await guanyarLoteria;
 console.log("Anem a sopar gràcies a que: " + plat);
}
sopar();

Pregunta 14: Què fa realment l'await quan troba una promesa?

1. Pausa l'execució: Para l'execució de la funció async actual en la línia exacta, esperant a que la promesa es resolgui o es rebutgi.
2. No bloqueja el fil principal: Encara que sembli que el codi es para, no para el fil principal del navegador o servidor.
3. Extreu el resultat: Si la promesa es fulfilled (es resol correctament), await extreu el resultat.
4. Llença errors: Si la promesa es rejected, await llença l'excepció directament, permetent al "try catch" actuar.

Pregunta 15: Com es gestionen els errors dins d'una funció async/await?

Es resolen envoltant el codi dins del "try catch". Quan la promesa falla await llença l'excepció i aquí actua el "catch".

async function getData() {
 try {
  const response = await fetch('https://api.example.com/data');
  const data = await response.json();
  console.log(data);
 } catch (error) {
  console.error('Error capturado:', error);
 }
}

Pregunta 16: Què passa si fas un await d'una funció que no retorna una promesa?

JavaScript ho torna automàticament com una promesa resolta, per fer-ho el codi pausa breument l'execució de la funció "async" per embolicar el valor. Però no causa errors, el resultat del "await" és el valor mateix. Això fa innecessari l'ús de l'"await" en aquests casos.

Pregunta 17: Per què async/await es considera millor que els .then() encadenats?

El principal motiu és la llegibilitat. També evita cadenes llargues i confuses, la gestió d'errors és més clara ja que el try catch és millor, i és ideal per a lògica complexa amb bucles o ifs.

// Amb .then():
fetch('/api/data')
 .then(res => res.json())
 .then(data => {
  return processData(data);
 })
 .then(result => {
  console.log(result);
 })
 .catch(err => console.error(err));

// Amb async/await:
try {
 const res = await fetch('/api/data');
 const data = await res.json();
 const result = await processData(data);
 console.log(result);
} catch (err) {
 console.error(err);
}

Pregunta 18: Es pot fer servir await fora d'una funció (Top-level await)? En quins casos?

Sí, sí que es pot, però en casos limitats: 1. Arxiu JS amb mòduls que contingui l'extensió ".mjs". 2. En navegadors moderns, utilitzant <script type="module">.

// archivo.mjs
const data = await fetch("https://api.example.com/data");
console.log(await data.json());

<script type="module">
 const res = await fetch("https://api.example.com");
 console.log(await res.json());
</script>

Pregunta 19: Com executaries dues funcions asíncrones en paral·lel dins d'una funció async?

Pots inicialitzar les funcions asíncrones abans de l'await perquè s'executin al mateix temps, o de manera més recomanada, utilitzant Promise.all.

// Forma correcta (paral·lel)
async function exemple() {
 const p1 = func1(); // comença
 const p2 = func2(); // comença també
 const a = await p1;
 const b = await p2;
 return [a, b];
}

// Forma més habitual: Promise.all
async function exemple() {
 const [a, b] = await Promise.all([
  func1(),
  func2()
 ]);
 return [a, b];
}

BLOC 3: Dades, REST API i Entorn

Pregunta 40: Escriu la sintaxi bàsica d'un bloc try...catch.

La declaració try…catch assenyala un bloc d’instruccions a intentar (try), i específica una resposta si es produeix una excepció (catch).

try {
 nonExistentFunction(); // Aquesta funció no existeix
} catch (error) {
 console.error("S'ha produït un error:", error.message);
}

Pregunta 41: Com comprovem si una petició fetch ha anat bé (codi 200) abans de llegir les dades? (response.ok)

Per comprovar si una petició fetch ha anat bé abans de llegir les dades, cal mirar la propietat response.ok (és true si l'estat és entre 200 i 299) o el response.status.

// Amb response.ok (millor pràctica):
fetch("products.json")
 .then((response) => {
  if (!response.ok) {
   throw new Error(`HTTP error: ${response.status}`);
  }
  return response.json();
 })
 .then((json) => console.log(json))
 .catch((err) => console.error(`Fetch problem: ${err.message}`));

// Amb response.status:
fetch('https://api.exemple.com/dades')
 .then(response => {
  if (response.status !== 200) {
   throw new Error(`Codi incorrecte: ${response.status}`);
  }
  return response.json();
 });

Pregunta 42: Per què necessitem un segon await quan fem response.json()?

Es necessita perquè és una operació asíncrona. fetch() retorna una Promise amb la resposta HTTP (les capçaleres), però el cos de la resposta pot trigar en descarregar-se completament. response.json() retorna una altra Promise que es resol un cop s'ha llegit i parsejat tot el cos a JSON.

async function obtenirDades() {
 // 1r await: Esperem a connectar i rebre les capçaleres
 const response = await fetch('https://api.exemple.com/dades');

 // 2n await: Esperem a descarregar el cos i convertir-lo a JSON
 const data = await response.json();
 console.log(data);
}

Pregunta 43: Com enviaries dades (mètode POST) amb fetch?

Has d’especificar diverses opcions dins d'un objecte com a segon paràmetre: el mètode HTTP ("POST"), les capçaleres indicant el format de la dada ("Content-Type"), i el cos (body) convertit a text amb JSON.stringify.

const jsonData = { nom: "Dani", rol: "Estudiant" };

const options = {
 method: "POST",
 headers: {
  "Content-Type": "application/json"
 },
 body: JSON.stringify(jsonData)
};

fetch("https://api.exemple.com/usuaris", options)
 .then(res => res.json())
 .then(data => console.log("Creat:", data));

Pregunta 44: Què són els "Headers" en una petició fetch?

Les capçaleres HTTP són informació addicional que s’envia juntament amb la petició per donar context sobre les dades (quin tipus de contingut enviem, tokens d'autenticació, etc.) o definir com s’han d’interpretar.

const headers = new Headers();
headers.set("Content-Type", "application/json");
headers.set("Authorization", "Bearer meu_token_secret");

// Recórrer i veure les capçaleres
for (let [key, value] of headers.entries()) {
 console.log(`Clau: ${key}, Valor: ${value}`);
}

Pregunta 45: Com podem cancel·lar una petició fetch que triga massa? (AbortController).

Es fa servir l’API AbortController. Ens dona un objecte 'signal' que passem a les opcions del fetch. Quan volem aturar-ho, cridem la funció controller.abort().

const controller = new AbortController();
const signal = controller.signal;

fetch('https://api.exemple.com/dades', { signal })
 .then(response => response.json())
 .then(data => console.log(data))
 .catch(error => {
  if (error.name === 'AbortError') {
   console.log('Petició cancel·lada per trigar massa');
  } else {
   console.error('Error:', error);
  }
});

// Cancel·lar la petició manualment (ex: als 3 segons)
setTimeout(() => controller.abort(), 3000);

Pregunta 46: Què vol dir JSON?

JSON vol dir JavaScript Object Notation, és un format basat en text estàndard per representar dades estructurades en la sintaxi d’objectes de JavaScript. És lleuger i fàcil d'enviar per xarxa.

// Exemple d'arxiu .json
{
 "usuari": "Aitor",
 "edat": 25,
 "actiu": true
}

Pregunta 47: Quina és la diferència principal entre un objecte JS i un JSON?

Un objecte JS és una estructura de dades funcional en memòria (pot tenir mètodes i funcions). Un JSON és purament un format de text d'intercanvi. Un JSON només pot tenir strings, ints, booleans, null, arrays i altres JSONs (les claus han d'anar entre cometes dobles).

// Objecte JS (Pot tenir funcions, i claus sense cometes)
const objJS = {
 nom: "Jesus",
 saludar: function() { console.log("Hola!"); }
};

// JSON (Només text estructurat i dades)
// '{"nom": "Jesus"}'

Pregunta 48: Com es converteix un String JSON en un objecte JS usable? (JSON.parse)

Per convertir una cadena de text JSON en un objecte utilitzable pel programa s’utilitza la funció JSON.parse(text).

const textJson = '{"marca":"Ford","model":"Mustang"}';
const cotxe = JSON.parse(textJson);

console.log(cotxe.marca); // Imprimeix: "Ford"

// Altres exemples de parse:
JSON.parse("true"); // boolean: true
JSON.parse('[1, 5, "fals"]'); // Array: [1, 5, "fals"]

Pregunta 49: Com es converteix un objecte JS en un String JSON per enviar-lo? (JSON.stringify)

Un objecte JS es converteix en format text JSON utilitzant el mètode JSON.stringify(objecte). És el pas necessari abans d'enviar dades amb un POST.

const usuari = {
 nom: "Dani",
 punts: 150
};

const dadesPerEnviar = JSON.stringify(usuari);
console.log(dadesPerEnviar);
// Imprimeix el text: '{"nom":"Dani","punts":150}'

Pregunta 50: Quins tipus de dades pot contenir un JSON?

Un JSON admet sis tipus de dades bàsiques: String (text), Number (números enters o decimals), Boolean (true/false), Array (llistes), Object (altres estructures clau/valor) i valors Null.

{
 "text": "Hola",
 "numero": 42.5,
 "boolea": false,
 "array": [1, 2, 3],
 "objecte": { "sub_clau": "valor" },
 "buit": null
}

Pregunta 51: Es poden posar funcions dins d'un fitxer JSON?

No es poden posar funcions dintre d’un fitxer JSON perquè només és un format de text dissenyat per a l'intercanvi de dades, no de lògica o comportament. Si intentes convertir una funció, s'ometrà.

const obj = {
 id: 1,
 acció: function() { return "Saltar"; }
};

console.log(JSON.stringify(obj));
// Resultat: '{"id":1}' (La funció desapareix)

console.log(JSON.stringify([new Number(3), new String("fals")]));
// Resultat: '[3,"fals"]'

Pregunta 52: Per què fem servir JSON i no XML avui dia?

Encara que la funció de tots dos és la mateixa (intercanvi de dades), JSON és més curt, menys pesat, i més fàcil de llegir. A més, es pot parsejar de manera nativa i molt ràpida a JavaScript amb JSON.parse(), sense necessitar llibreries pesades.

// Format JSON (Ràpid i net)
{ "usuari": "Aitor", "edat": 30 }

<!-- Format XML (Més verbós i feixuc) -->
<usuari>
 <nom>Aitor</nom>
 <edat>30</edat>
</usuari>

Pregunta 53: Quina és la principal diferència entre l'Event Loop del navegador i el de Node.js?

El navegador prioritza la renderització de la interfície gràfica i la interacció amb l'usuari. En canvi, Node.js se centra en operacions I/O (entrada/sortida, com llegir fitxers del disc o bases de dades) utilitzant una llibreria pròpia anomenada Libuv i un "pool de fils" per fons.

// Al navegador, l'Event loop gestiona el DOM
document.getElementById('btn').addEventListener('click', ...);

// A Node.js, l'Event loop està dissenyat per gestionar el sistema (I/O)
const fs = require('fs');
fs.readFile('arxiu.txt', (err, data) => {
 console.log("Arxiu llegit");
});

Pregunta 54: El fetch és natiu en Node.js? Des de quina versió?

Sí, és natiu des de la versió 18.0.0. Abans calia instal·lar paquets de tercers com 'node-fetch' o 'axios', però ara ja funciona igual que al navegador.

// En Node >= 18, això ja no dona error de "fetch is not defined"
async function obtenirUsuarisServer() {
 const res = await fetch('https://api.github.com/users');
 const data = await res.json();
 console.log(data);
}

Pregunta 55: Si fas un fetch a una API des del navegador i et dona un error de "CORS", què està passant?

Els CORS (Cross-Origin Resource Sharing) és un mecanisme de seguretat dels navegadors que bloqueja peticions web cap a un domini diferent de l'actual. Si l'API no té configurades les capçaleres per donar permís al teu domini, el navegador bloquejarà la resposta.

// Error típic que apareix a la consola del navegador:
// Access to fetch at 'https://api.altresite.com' from origin 'http://localhost:5500'
// has been blocked by CORS policy.

// Solució: El servidor ha de retornar aquesta capçalera:
// Access-Control-Allow-Origin: *

Pregunta 56: Què signifiquen les sigles API i quina és la seva funció principal en el desenvolupament web?

Significa Application Programming Interface (Interfície de Programació d'Aplicacions). La seva funció és fer de pont de connexió per permetre que dues aplicacions independents es comuniquin i intercanviïn dades de forma automatitzada (ex: La teva web consumint dades de l'API del Temps).

// La nostra aplicació JS parla amb l'aplicació de GitHub via API
fetch('https://api.github.com/users/octocat')
 .then(res => res.json())
 .then(usuari => console.log(`Repos de l'usuari: ${usuari.public_repos}`));

Pregunta 57: Què fa que una API sigui considerada "REST"?

REST significa "Representational State Transfer". Una API es considera RESTful quan segueix uns principis arquitectònics com operar sobre recursos identificats per URLs uniformes, ser "stateless" (cada petició és independent) i utilitzar els mètodes HTTP estàndard (GET, POST, PUT, DELETE).

// Rutes clàssiques i predictibles tipus REST:
// GET /api/usuaris -> Obté llista d'usuaris
// GET /api/usuaris/123 -> Obté detalls de l'usuari 123
// POST /api/usuaris -> Crea un usuari nou
// DELETE /api/usuaris/123 -> Esborra l'usuari 123

Pregunta 58: Relaciona els mètodes HTTP clàssics (GET, POST, PUT, DELETE) amb les accions de base de dades (Llegir, Crear, Actualitzar, Esborrar).

  • GET: Llegir (Read)
  • POST: Crear (Create)
  • PUT / PATCH: Actualitzar (Update)
  • DELETE: Esborrar (Delete)
Són les famoses operacions CRUD.

// Llegir (GET)
fetch('/api/notes');

// Crear (POST)
fetch('/api/notes', { method: 'POST', body: JSON.stringify(novaNota) });

// Actualitzar (PUT)
fetch('/api/notes/5', { method: 'PUT', body: JSON.stringify(canvisNota) });

// Esborrar (DELETE)
fetch('/api/notes/5', { method: 'DELETE' });

Pregunta 59: Si fem una petició GET a la URL https://api.botiga.com/productes/5, què estem demanant exactament al servidor?

Estem demanant que ens retorni la informació detallada d'un recurs concret: específicament el producte que tingui com a identificador únic (ID) el número 5.

fetch('https://api.botiga.com/productes/5')
 .then(res => res.json())
 .then(producte => {
  console.log("Nom del producte 5:", producte.nom);
  console.log("Preu del producte 5:", producte.preu);
 });
